## КРИТИЧНО: Workflow перед любой правкой

1. **Скачивать `index.html` ТОЛЬКО через GitHub API**, никогда через raw.githubusercontent.com
   (у него CDN-кеш на несколько минут — можно получить старую версию и затереть свежий фикс):
   ```python
   GET https://api.github.com/repos/{repo}/contents/index.html
   # content в base64, декодировать
   ```
2. **После каждого редактирования CSS — проверять баланс скобок** перед пушем:
   ```python
   css.count('{') == css.count('}')
   ```
   Особенно когда правка — замена одной строки на другую без явного `}`.
3. **Рендерить реальным браузером (Playwright) перед пушем**, а не гадать по коду:
   - проверять `document.documentElement.scrollWidth - clientWidth` (должно быть 0, иначе обрезка)
   - делать скриншоты на нескольких ширинах: 390 (моб), 768, 900, 1078/1280, 1440, 1820
4. Текст в HTML содержит `\xa0` (неразрывные пробелы) между некоторыми словами —
   `str_replace` с обычным пробелом не сматчится. Редактировать такие строки через
   Python по номеру строки (`lines[N] = ...`), а не str_replace с обычным текстом.

## Типографика

### Висячие предлоги
Короткие предлоги/союзы/частицы не должны оставаться в конце строки — после них `&nbsp;`.
Список: в, во, на, с, со, к, ко, у, о, об, обо, и, а, но, из, по, за, до, от, не, ни, при,
для, без, над, под, про, через, или, да, уж, уже, ещё.

Проверочный скрипт (запускать после любых текстовых правок):
```python
prepositions = ['в','во','на','с','со','к','ко','у','о','об','обо','и','а','но','из','по',
'за','до','от','не','ни','при','для','без','над','под','про','через','или','да','уж','уже','ещё']
pattern = r'(?<![a-zA-Zа-яёА-ЯЁ&\d])(' + '|'.join(prepositions) + r')[ \t\n]+(?=[a-zA-Zа-яёА-ЯЁ])'
# искать matches только в текстовых нодах (не внутри <style>/<script>)
```

### Тире и кавычки
- Длинное тире `—` (не дефис), с `&nbsp;` вокруг: `&nbsp;—&nbsp;`
- Русские кавычки «ёлочки», не прямые `"`

## Ширина блоков (важно!)
Контейнер `.container` = 1280px (стандарт под ноутбуки/Mac).
Внутренние блоки расширены до **1080-1150px** (не 820, не 980!) — иначе текст выглядит
гораздо уже героя и блоки кажутся несогласованными. Если добавляешь новый текстовый/карточный
блок — ширина должна быть в этом диапазоне, а не дефолтные узкие 600-900px.

## Секция "5 шагов" — динамический коннектор (важно!)

Линия между кружками 1→2→3→4→5 строится **JavaScript'ом по реальным координатам**,
а не хардкодом в процентах (хардкод сломался при смене ширины экрана — текст и линия
не совпадали при разных размерах текста/окна).

Функция `buildStepsPath()` в `<script>`:
- Меряет `getBoundingClientRect()` кружков и текстовых блоков
- Строит путь по правилу: **горизонтальные движения только на высоте кружков**
  (там нет текста), **вертикальные — только в безопасном коридоре** между колонками
  (`gapX`, между правым краем левой колонки и левым краем правой)
- Текущий маршрут: 1→2 (прямая), 2→3 (огибает блок 2 справа снаружи, идёт над 3 и 4,
  заходит в 3 слева), 3→4 (прямая), 4→5 (огибает блок 4 справа снаружи, заходит в 5 сверху)
- `stroke-dasharray`/`stroke-dashoffset` для анимации отрисовки — длина считается
  динамически через `path.getTotalLength()`, **не хардкодить число** (один раз сломалось:
  захардкоженные 3000 оказались меньше реальной длины 3431, конец линии не дорисовывался)
- **Важно**: построение пути отложено на 1150мс после появления секции в зоне видимости —
  это ровно столько, сколько длится анимация появления карточек (`.step-card`). Если измерить
  позиции раньше — кружки ещё в смещённом (translateY) состоянии анимации, координаты будут
  неверные на ~30px.
- На мобильной (`≤768px`) линия скрыта (`.steps-path-svg{display:none}`) — там вёрстка
  в один столбец, линия не нужна.

При любой правке этой секции — обязательно рендерить реальным браузером и проверять
визуально (через `view` tool на скриншот), что линия не пересекает текст ни на одной
ширине экрана.

## Фото Ольги (`olga.svg`) в круге — `object-position`

Текущее значение: `object-position: 50% 0%;`

**Направление (проверено эмпирически, не доверять интуиции):**
- **Меньший процент** Y → видно ВЕРХ фото → больше отступа над головой, голова целиком видна
- **Больший процент** Y → видно НИЗ фото → голову обрезает круг

Если попросят "опустить фото" в смысле "убрать обрезку головы" — **уменьшать** процент.
Если попросят буквально "сдвинуть видимую область вниз" — **увеличивать** процент,
но это будет резать голову сильнее (на 30% уже заметная обрезка).
0% — практически предел в сторону "больше головы видно", дальше только увеличивать сам круг.

## Герой (первый блок) — картинка девушки (`girl.svg`)

- `.hero-right img`: `max-height: 620px` (десктоп), `360px` (мобайл)
- `margin-top: 150px` — опускает картинку вниз
- `margin-bottom: -46px` — **обязательно через отрицательный margin**, не через
  увеличение margin-top, чтобы прижать низ картинки к границе секции: контейнер `.hero`
  авто-растёт под контент (`min-height` — это пол, не потолок), поэтому простое увеличение
  margin-top НЕ уменьшает зазор до низа секции — секция просто становится выше.
  Только отрицательный margin-bottom компенсирует `padding-bottom` секции (`40px` → нужно
  гасить ~46px с учётом float-анимации).
